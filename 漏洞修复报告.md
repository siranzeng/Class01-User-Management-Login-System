## v2.4 — 个人中心/充值 + 越权&业务逻辑漏洞修复

**修复日期：** 2026-07-22  
**修复人员：** 曾思填  
**审计范围：** /profile 和 /recharge 路由的业务逻辑及权限校验  
**审计方式：** 白盒代码审计 + 黑盒渗透测试

### 1. /profile 越权访问（高危）

**问题：** /profile 路由完全没有 session 登录校验和权限校验。攻击者无需登录即可通过修改 URL 参数 `user_id` 遍历所有用户的个人资料（含邮箱、手机、余额）。

**漏洞代码（修复前）：** `app.py` 第 386-409 行

```python
@app.route("/profile")
def profile():
    user_id = request.args.get("user_id", "")
    user = None
    if user_id and user_id.isdigit():
        conn = sqlite3.connect("data/users.db")
        c = conn.cursor()
        c.execute("SELECT id, username, email, phone FROM users WHERE id = ?", (user_id,))
        row = c.fetchone()
        conn.close()
        if row:
            uid, uname, email, phone = row
            user_data = USERS.get(uname)
            if user_data:
                user = {
                    "id": uid, "username": uname,
                    "email": email, "phone": phone,
                    "balance": user_data.get("balance", 0),
                    "role": user_data.get("role", "user")
                }
    return render_template("profile.html", user=user)
```

**修复：** 增加 session 登录校验，并增加权限判断：普通用户只能查看自己的资料，管理员可查看全部。

```python
if "username" not in session:
    return redirect("/login")

target_uid = int(user_id)
if target_uid != current_data["id"] and current_data.get("role") != "admin":
    error = "无权查看其他用户的资料"
```

**POC 验证结果：**

| 测试场景 | 修复前 | 修复后 |
|----------|--------|--------|
| 未登录访问 /profile | 正常返回任意用户资料 | 302 重定向到 /login |
| alice 查看 admin 资料 | 正常返回 admin 信息 | 提示"无权查看其他用户的资料" |
| admin 查看 alice 资料 | 正常返回 alice 信息 | 正常返回（管理员权限） |
| 遍历 user_id=1~100 | 批量枚举全部用户信息 | 非管理员被拒绝 |

### 2. /recharge 越权操作（高危）

**问题：** /recharge 路由完全没有 session 校验和权限校验。攻击者无需登录即可通过构造 POST 请求给任意用户充值（或通过负数金额盗刷余额）。

**漏洞代码（修复前）：** `app.py` 第 412-429 行

```python
@app.route("/recharge", methods=["POST"])
def recharge():
    user_id = request.form.get("user_id", "")
    amount = request.form.get("amount", "0")
    if user_id and user_id.isdigit():
        conn = sqlite3.connect("data/users.db")
        c = conn.cursor()
        c.execute("SELECT username FROM users WHERE id = ?", (user_id,))
        row = c.fetchone()
        conn.close()
        if row:
            username = row[0]
            if username in USERS:
                USERS[username]["balance"] += float(amount)
    return redirect(f"/profile?user_id={user_id}")
```

**修复：** 增加 session 登录校验，并限制非管理员只能给自己充值。

```python
if "username" not in session:
    return redirect("/login")

target_uid = int(user_id)
if target_uid != current_data["id"] and current_data.get("role") != "admin":
    error = "无权操作其他用户的充值"
```

**POC 验证结果：**

| 测试场景 | 修复前 | 修复后 |
|----------|--------|--------|
| 未登录充值 | 成功充值 | 302 重定向到 /login |
| alice 给 admin 充值 | 成功增加 admin 余额 | 拒绝并提示错误 |
| admin 给 alice 充值 | 成功增加 alice 余额 | 成功（管理员权限） |
| 批量遍历充值 user_id=1~100 | 全部成功 | 非管理员被拒绝 |

### 3. 负数金额盗刷余额（高危）

**问题：** amount 参数未做任何正负校验，攻击者可传入负数值减少他人余额（盗刷），或传入极大值使余额溢出。

**漏洞代码（修复前）：** `app.py` 第 416、427 行

```python
amount = request.form.get("amount", "0")      # 无正负校验
USERS[username]["balance"] += float(amount)    # 直接累加
```

**修复：** 增加 amount 正数校验，拒绝非正数值。

```python
try:
    amount_val = float(amount)
except ValueError:
    amount_val = 0

if amount_val <= 0:
    error = "金额必须为正数"
    return redirect(f"/profile?user_id={user_id}&error={error}")
```

**POC 验证结果：**

| 测试场景 | 修复前 | 修复后 |
|----------|--------|--------|
| amount=-500 负数充值 | 余额成功扣减 500 | 拒绝："金额必须为正数" |
| amount=0 零充值 | 余额不变但无提示 | 拒绝："金额必须为正数" |
| amount=abc 非法输入 | ValueError 500 错误 | 拒绝："金额必须为正数" |
| amount=200 正常充值 | 余额 +200 | 余额 +200 |

### 4. 导航栏硬编码越权链接（中危）

**问题：** base.html 导航栏"个人中心"链接硬编码为 `/profile?user_id=1`，导致所有用户点击后默认查看管理员资料。

**漏洞代码（修复前）：** `templates/base.html`

```html
<a href="/profile?user_id=1" class="navbar-link">个人中心</a>
```

**修复：** 改为 `/profile`（不带参数），路由自动使用当前登录用户 ID。

```html
<a href="/profile" class="navbar-link">个人中心</a>
```

**修改文件：** app.py, templates/profile.html(新增), templates/base.html, templates/index.html, README.md

## v2.3 — 文件上传安全加固

**修复日期：** 2026-07-21  
**修复人员：** 曾思填

### 1. 路径穿越攻击（高危）

**问题：** 上传文件名中包含 `../` 等路径穿越字符时，可写入 `/static/uploads/` 上级目录，覆盖系统文件。

**漏洞代码（修复前）：**

```python
file.save(os.path.join("static/uploads", file.filename))
# 文件名 "../../etc/passwd" 可穿越到上级目录
```

**修复：** 过滤 `../` 和 `/` 开头等路径穿越字符。

```python
raw_name = file.filename.replace("\\", "/")
if ".." in raw_name or raw_name.startswith("/"):
    error = "文件名不合法"
```

### 2. 可执行文件上传（高危）

**问题：** 无后缀白名单，可上传 `.php`、`.py`、`.exe`、`.sh` 等可执行脚本到服务器。

**修复：** 后缀白名单仅允许图片格式。

```python
ALLOWED_EXT = {"jpg", "jpeg", "png", "gif", "bmp", "webp"}
ext = raw_name.rsplit(".", 1)[-1].lower() if "." in raw_name else ""
if ext not in ALLOWED_EXT:
    error = "仅支持图片格式"
```

### 3. MIME 类型伪造（中危）

**问题：** 仅校验后缀，攻击者可上传含恶意代码的图片（图片马）。

**修复：** 校验 Content-Type 字段。

```python
ALLOWED_MIME = {"image/jpeg", "image/png", "image/gif", "image/bmp", "image/webp"}
mime = file.content_type
if mime not in ALLOWED_MIME:
    error = "文件类型不合法"
```

### 4. 图片马/隐藏代码（中危）

**问题：** 攻击者可在合法后缀和 MIME 的文件中嵌入 PHP/JS 代码。

**修复：** 读取文件头魔数验证真实类型。

```python
def check_image_magic(data):
    if data[0:2] == b"\xff\xd8": return True   # JPEG
    if data[0:4] == b"\x89PNG": return True    # PNG
    if data[0:6] in (b"GIF87a", b"GIF89a"): return True
    if data[0:2] == b"BM": return True         # BMP
    if data[0:4] == b"RIFF" and data[8:12] == b"WEBP": return True
    return False
```

### 5. 文件名冲突覆盖（低危）

**问题：** 同名文件会互相覆盖，导致旧文件丢失或被恶意替换。

**修复：** UUID+时间戳重命名保存。

```python
saved_name = f"{secrets.token_hex(8)}_{int(time.time())}.{ext}"
file.save(os.path.join("static/uploads", saved_name))
```

**修改文件：** app.py, .gitignore, templates/upload.html, README.md

## v2.1 — SQL 注入修复 + 注册/搜索功能

**修复日期：** 2026-07-20  
**修复人员：** 曾思填

### 1. SQL 注入漏洞（高危）

**问题：** 注册和搜索功能中用 f-string 拼接 SQL，用户输入未做任何过滤。攻击者可通过 UNION 注入获取任意数据，通过 OR 注入遍历所有用户。

**漏洞代码（修复前）：**

```python
# 搜索 - 字符串拼接
sql = f"SELECT ... WHERE username LIKE '%{keyword}%' OR email LIKE '%{keyword}%'"
c.execute(sql)

# 注册 - 字符串拼接
sql = f"INSERT INTO users ... VALUES ('{username}', '{password}', '{email}', '{phone}')"
c.execute(sql)
```

**修复：** 改用参数化查询，用户输入通过 `?` 占位符传入，不参与 SQL 语句拼接。

```python
sql = "SELECT ... WHERE username LIKE ? OR email LIKE ?"
c.execute(sql, (f"%{keyword}%", f"%{keyword}%"))

sql = "INSERT INTO users ... VALUES (?, ?, ?, ?)"
c.execute(sql, (username, password, email, phone))
```

**POC 验证结果：**

| 注入方式 | 修复前 | 修复后 |
|----------|--------|--------|
| `' UNION SELECT 1,'inj',...` | 返回自定义数据 inj | 无搜索结果 |
| `' OR '1'='1` | 返回所有用户 | 无搜索结果 |
| 注册注入 `hacker',+'pass'...` | SQL 被篡改 | 字符串原样存储 |

### 2. 新增功能

- 用户注册（/register 路由），注册后跳转登录页
- 用户搜索（首页搜索框 + /search API），按用户名或邮箱模糊查询

**修改文件：** app.py, templates/register.html(新增), templates/index.html, templates/base.html, templates/login.html, static/css/style.css

## v2.0 — 安全加固（防暴力破解 + 密码保护）

**修复日期：** 2026-07-19  
**修复人员：** 曾思填

### 1. 明文密码存储（高危）

**问题：** 密码以明文存储在 `USERS` 字典和 SQLite 数据库中，数据库文件泄露即所有账号失陷。

**漏洞代码（修复前）：**

```python
USERS = {
    "admin": {"password": "admin123", ...},
    "alice": {"password": "alice2025", ...}
}
```

**修复：** 使用 `werkzeug.security.generate_password_hash` 进行 PBKDF2+SHA256 哈希存储。

```python
from werkzeug.security import generate_password_hash
USERS = {
    "admin": {"password": generate_password_hash("admin123"), ...},
    "alice": {"password": generate_password_hash("alice2025"), ...}
}
```

### 2. 无 CSRF 防护（高危）

**问题：** 登录表单无 CSRF Token，攻击者可构造跨站请求伪造登录。

**修复：** 一次性 Token + `secrets.compare_digest` 校验。

```python
def generate_csrf_token():
    if "csrf_token" not in session:
        session["csrf_token"] = secrets.token_hex(32)
    return session["csrf_token"]

def check_csrf_token():
    token = request.form.get("csrf_token", "")
    stored = session.get("csrf_token")
    if not stored or not secrets.compare_digest(stored, token):
        return False
    session["csrf_token"] = secrets.token_hex(32)
    return True
```

### 3. 暴力破解无防护（高危）

**问题：** 登录接口无任何频率限制，攻击者可无限次尝试密码。

**修复：** 四层防护：

| 防护层 | 参数 |
|--------|------|
| IP 限流 | 每分钟每个 IP 最多 10 次请求 |
| 账户锁定 | 连续 5 次失败锁定 15 分钟 |
| 指数退避 | 1s → 2s → 4s → ... → 30s（上限） |
| 用户枚举防护 | 统一错误"用户名或密码错误" |

### 4. 密码前端明文展示（中危）

**问题：** index.html 首页直接显示用户明文密码。

```html
<li>密码：admin123</li>
```

**修复：** 改为显示 `******`。

```html
<li><span>密码：</span>******</li>
```

### 5. HTML 注释泄露账号（中危）

**问题：** HTML 注释中包含测试账号明文密码。

```html
<!-- 测试账号: admin/admin123 -->
```

**修复：** 注释中移除明文密码，仅保留用户名。

### 6. SECRET_KEY 硬编码（中危）

**问题：** `app.secret_key` 设置为固定字符串，可被伪造 session。

**修复：** 使用 `secrets.token_hex(32)` 随机生成。

```python
app.secret_key = secrets.token_hex(32)
```

**修改文件：** app.py(重写), templates/login.html, templates/index.html, 漏洞修复报告.md(新增), README.md
